## 1. Принципы антихрупкости

### 1.1. Что такое антихрупкость в контексте системы

**Хрупкая система:** сломается при стрессе (пример: монолит на одном сервере)  
**Устойчивая система:** выдержит стресс без изменений (пример: реплицированная БД)  
**Антихрупкая система:** становится сильнее от стресса (пример: система с автоматическим масштабированием и обучением на ошибках)

### 1.2. Наши принципы

#### Принцип №1: Decoupling (Разделение)
- Каждый модуль работает независимо
- Падение одного модуля не роняет систему
- Асинхронная коммуникация через очереди

#### Принцип №2: Redundancy (Избыточность)
- Критические компоненты задублированы
- Несколько путей выполнения задачи
- Fallback-механизмы на всех уровнях

#### Принцип №3: Graceful Degradation (Плавная деградация)
- Система работает с урезанным функционалом при сбоях
- Нет "всё или ничего"
- Пользователь всегда получает ответ (даже если "попробуйте позже")

#### Принцип №4: Observable (Наблюдаемость)
- Все процессы логируются
- Метрики в реальном времени
- Алерты при аномалиях
- Возможность воспроизвести любую операцию

#### Принцип №5: Self-Healing (Самовосстановление)
- Автоматический рестарт упавших сервисов
- Автоматическое переключение на резервные каналы
- Очистка зависших задач

#### Принцип №6: Data Integrity (Целостность данных)
- Все критические операции атомарны
- Транзакции с откатом при ошибках
- Регулярные бэкапы с проверкой восстановления

### 2.2. Стек технологий

#### Оркестрация: n8n Cloud
- **Почему n8n:** визуальный workflow builder, 400+ готовых интеграций, webhook-friendly, self-hosted или cloud
- **Роль:** главный оркестратор всех процессов
- **Антихрупкость:** автоматический retry, error workflows, паузы и возобновления

#### Бэкенд: Python (FastAPI)
- **Почему Python + FastAPI:** быстрая разработка, огромная экосистема AI/ML, отличная производительность
- **Роль:** тяжелая бизнес-логика, AI-модели, обработка данных
- **Компоненты:**
  - `core/` — ядро системы (конфиги, логирование, базовые классы)
  - `services/` — бизнес-сервисы (CRM, телефония, документы)
  - `integrations/` — адаптеры для внешних систем (1С, VoIP, мессенджеры)
  - `ai/` — AI-модули (Гермес, обработка текста, аналитика)
  - `api/` — REST API endpoints
  - `workers/` — фоновые задачи (Celery/RQ)

#### База данных: PostgreSQL + Redis
- **PostgreSQL:** основная БД (пациенты, записи, история, документы)
- **Redis:** кэш, очереди задач, сессии, rate limiting
- **Антихрупкость:** 
  - PostgreSQL: streaming replication (master-replica), point-in-time recovery
  - Redis: persistence (AOF + RDB), sentinel для автоматического failover

#### Хранилище файлов: MinIO (S3-compatible)
- **Почему MinIO:** self-hosted S3, дешевле облаков, полный контроль
- **Роль:** документы пациентов, медицинские записи, аудиозаписи звонков
- **Антихрупкость:** erasure coding, версионирование, репликация

#### Очереди: Celery + Redis
- **Роль:** асинхронные задачи (звонки, рассылки, обработка данных)
- **Антихрупкость:** retry с экспоненциальным backoff, dead letter queue, priority queues

### 3.2. Режимы работы Гермеса

#### Режим 1: Служба заботы для пациентов

**Каналы:**
- Telegram (основной)
- WhatsApp
- Голосовые звонки (через TTS/STT, БЕЗ клонирования голоса Мадякина)
- Веб-чат на сайте
- VK/Instagram direct (через API или веб-хуки)

**Сценарии:**

**A. Подтверждение записи (за 24 часа)**
```
Триггер: Запись в 1С + время "завтра в 14:30"
Действие: 
  1. Гермес отправляет сообщение в Telegram
     "Добрый день! Напоминаем, что завтра в 14:30 вас ждёт приём 
      у доктора Мадякина по адресу Салимжанова 21. Подтвердите, 
      пожалуйста: ДА / Перенести / Отменить"
  2. Если нет ответа через 6 часов → звонок
  3. Если не подтвердил → передача администратору
```

**B. Follow-up после приёма (через 2-3 дня)**
```
Триггер: Завершенный прием в 1С
Действие:
  1. Гермес отправляет сообщение:
     "Здравствуйте! Как ваше самочувствие после приёма у доктора 
      Мадякина? Оцените от 1 до 10."
  2. Сбор фидбэка (текст или голосовое)
  3. Рассказ о последствиях (контекстно под диагноз):
     - Что НОРМАЛЬНО испытывать (лёгкая боль, усталость...)
     - На что обратить ВНИМАНИЕ (тревожные симптомы)
     - Когда прийти на повторный приём
  4. Закрытие на курс (если был разовый прием):
     "Доктор рекомендовал курс из 4 сеансов. Записать вас?"
  5. Запись NPS-оценки + текста в БД
  6. Если оценка ≤6 → алерт руководству + звонок администратора
```

**C. Возврат no-show (через 1 час после пропуска)**
```
Триггер: Запись в 1С + время прошло + пациент не пришёл
Действие:
  1. Сообщение: "Вы были записаны на 14:30, но не смогли прийти. 
     Всё в порядке? Перенести на другое время?"
  2. Быстрая перезапись на ближайшее окно
  3. Если не отвечает → звонок администратора на следующий день
```

**D. Реактивация "спящих" пациентов**
```
Триггер: Нет визитов 6+ месяцев + LTV >50K (из RFM-сегментации)
Действие:
  1. Персонализированное сообщение под сегмент:
     - RiskWhale (6-12 мес): "Давно не виделись! Как дела со 
       здоровьем? Доктор Мадякин приглашает на бесплатную 
       контрольную диагностику"
     - SleepingWhale (1-2 года): "Прошло больше года... 
       Бесплатный осмотр + видеоразбор от Мадякина"
     - LostWhale (2+ года): "Помните, как мы работали над 
       [диагноз]? Хотим узнать, как сейчас"
  2. Запись на прием через Гермеса
  3. Если не отвечает → в очередь для звонка администратором
```

**E. Сбор отзывов на Я.Карты / 2ГИС / ProDoctors**
```
Триггер: Завершенный курс + NPS ≥9
Действие:
  1. "Спасибо за доверие! Если вам всё понравилось, могли бы 
     оставить отзыв на [платформа]? Это очень поможет другим 
     пациентам. Вот ссылка: [url]"
  2. Через 3 дня → проверка, оставил ли отзыв
  3. Если оставил → благодарность + бонус (скидка на следующий курс)
```

**F. День рождения пациента**
```
Триггер: ДР из 1С
Действие:
  1. Поздравление + персональная акция:
     "С днём рождения! Дарим 20% скидку на любую услугу 
      до конца месяца. Записаться?"
```

**G. Триггерные акции по сегментам**
```
Триггер: Новая акция в клинике (например, "Беременность без боли")
Действие:
  1. Фильтр целевой аудитории (беременные, 28-40 недель)
  2. Рассылка персонализированного предложения
  3. Запись через Гермеса
```

### 3.3. Технические детали Гермеса

#### Стек AI
- **LangChain** — фреймворк для LLM-приложений
- **Claude Sonnet 4.5** — основная модель (рассуждения, медицинский контекст)
- **GPT-4 Turbo** — fallback + специфические задачи (JSON generation)
- **Gemini Pro** — дополнительная модель для экспериментов
- **Whisper** — распознавание речи (если голосовые звонки)
- **ElevenLabs / Yandex SpeechKit** — синтез речи (стандартный голос, НЕ Мадякина)

#### Vector DB для RAG
- **ChromaDB** или **Qdrant** — хранилище эмбеддингов
- **Контент:**
  - Медицинские протоколы клиники
  - FAQ
  - История успешных кейсов
  - Инструкции для сотрудников

#### Context Management
```python
class GermesContext:
    """Контекст разговора с Гермесом"""
    
    session_id: str  # ID сессии (уникален для каждого разговора)
    user_id: str  # ID пациента или сотрудника
    user_type: str  # "patient" или "employee"
    
    # Краткосрочная память (текущий разговор)
    conversation_history: List[Message]
    
    # Долгосрочная память (из 1С)
    patient_profile: Optional[PatientProfile]
    medical_history: List[MedicalRecord]
    
    # Контекст
    current_intent: str  # "booking", "feedback", "reactivation"...
    entities: Dict  # извлечённые сущности (дата, время, врач...)
    
    # Состояние
    state: str  # "awaiting_confirmation", "collecting_feedback"...
    pending_actions: List[Action]  # отложенные действия
```

#### Антихрупкость Гермеса

**Fallback-цепочка моделей:**
```
Claude Sonnet 4.5 (primary)
    ↓ (если ошибка или таймаут)
GPT-4 Turbo (fallback)
    ↓ (если ошибка)
Gemini Pro (fallback)
    ↓ (если всё упало)
Заготовленный ответ + передача администратору
```

**Retry-логика:**
- 3 попытки с экспоненциальным backoff
- Кэширование успешных ответов (для идентичных запросов)
- Graceful degradation (если AI недоступен, переключение на rule-based логику)

**Rate limiting:**
- Ограничение запросов на пациента (не более 10 сообщений/час)
- Антиспам-фильтры
- Приоритизация критических задач (подтверждение записи > поздравление с ДР)

    created_by UUID REFERENCES users(id),
    updated_at TIMESTAMP,
    updated_by UUID REFERENCES users(id),
    deleted_at TIMESTAMP,
    
    -- Индексы для поиска
    search_vector tsvector GENERATED ALWAYS AS (
        to_tsvector('russian', coalesce(title, '') || ' ' || coalesce(description, ''))
    ) STORED
);

CREATE INDEX idx_documents_patient ON documents(patient_id);
CREATE INDEX idx_documents_type ON documents(type);

CREATE INDEX idx_documents_search ON documents USING GIN(search_vector);

### 4.4. Интеграция с 1С

**Синхронизация медкарт:**
- n8n workflow с расписанием каждые 30 минут
- Проверка изменений в 1С (новые записи, обновления)
- Скачивание документов через API 1С
- Загрузка в MinIO + запись метаданных в PostgreSQL

**Двусторонняя синхронизация:**
- Документы, загруженные пациентом → автоматически добавляются в 1С
- Протоколы приёмов из 1С → автоматически доступны в EDM

### 4.5. Антихрупкость EDM

**Версионирование:**
- Каждое изменение документа создаёт новую версию
- Старые версии остаются в MinIO (с пометкой версии)
- Возможность откатиться к любой версии

**Резервное копирование:**
- Ежедневный бэкап MinIO → внешнее хранилище (S3 / rsync.net)
- Ежедневный бэкап PostgreSQL (metaданные)
- Хранение бэкапов 90 дней

**Аудит:**
- Все операции логируются (кто, когда, что)
- Невозможно удалить документ без следа (soft delete)
- История изменений доступна администраторам

#### Стратегия синхронизации

**Режим 1: Реактивная синхронизация (webhooks)**
- 1С отправляет webhook при создании/изменении записи
- Strong AI OS обрабатывает webhook и обновляет свою БД
- **Проблема:** не все версии 1С поддерживают webhooks

**Режим 2: Polling (опрос по расписанию)**
- n8n workflow каждые 5 минут опрашивает 1С на наличие изменений
- Проверка по полю "Дата изменения" (ДатаИзменения)
- Загрузка только изменённых/новых записей
- **Используем этот режим** как основной

**Режим 3: Гибридный**
- Webhooks для критических событий (новая запись на приём)
- Polling для всего остального
- **Рекомендуется** при наличии поддержки webhooks в 1С

#### Антихрупкость интеграции с 1С

**Кэширование:**
```python
# Структура кэша в Redis
patients:{patient_id}  # данные пациента, TTL 1 час
schedule:{doctor_id}:{date}  # расписание врача, TTL 30 минут
appointments:{appointment_id}  # запись на приём, TTL 10 минут
```

**Fallback при недоступности 1С:**
- Если 1С недоступен, система работает на кэшированных данных
- Пользователь видит предупреждение: "Данные могут быть неактуальны"
- Запросы на изменение (запись на приём) попадают в очередь
- При восстановлении 1С → автоматическая синхронизация очереди

**Rate limiting:**
- Не более 100 запросов/минуту к 1С
- Батчинг запросов (группировка)
- Приоритизация критических запросов

### 6.2. n8n Workflows (детальные сценарии)

#### Workflow 1: Обработка нового лида

```
Триггер: Webhook от Avito / ProDoctors / Лендинга
    ↓
Validation (проверка данных лида)
    ↓
Дедупликация (проверка, нет ли дубля в БД)
    ↓
Создание записи в PostgreSQL
    ↓
┌─────────────────────────────────┐
│  Отправка в Гермеса             │
│  Задача: "Первый контакт с лидом"│
└─────────────────────────────────┘
    ↓
Ожидание ответа (timeout 2 часа)
    ↓
IF (лид ответил):
    ↓
    Запись на приём → 1С API
    ↓
    Подтверждение лиду
    ↓
    Уведомление админу
ELSE:
    ↓
    Передача администратору (звонок)
```

#### Workflow 2: Подтверждение записи

```
Триггер: Запланированная задача (каждые 30 минут)
    ↓
Запрос к PostgreSQL:
    SELECT * FROM appointments
    WHERE date = TOMORROW
    AND status = 'scheduled'
    AND confirmed = FALSE
    ↓
Для каждой записи:
    ↓
    Отправка через Гермеса (Telegram / WhatsApp)
    ↓
    Ожидание подтверждения (6 часов)
    ↓
    IF (подтверждено):
        ↓
        UPDATE appointments SET confirmed = TRUE
    ELSE IF (перенос):
        ↓
        Поиск свободного времени → Новая запись
    ELSE IF (отмена):
        ↓
        UPDATE appointments SET status = 'cancelled'
    ELSE (нет ответа):
        ↓
        Звонок через VoIP / Уведомление админу
```

#### Workflow 3: Follow-up после приёма

```
Триггер: Запланированная задача (каждые 1 час)
    ↓
Запрос к PostgreSQL:
    SELECT * FROM appointments
    WHERE status = 'completed'
    AND completed_at BETWEEN NOW() - INTERVAL '3 days' AND NOW() - INTERVAL '2 days'
    AND followup_sent = FALSE
    ↓
Для каждого приёма:
    ↓
    Получение контекста пациента:
        - Диагноз
        - Назначенное лечение
        - История приёмов
    ↓
    Генерация персонализированного сообщения через Гермеса:
        "Здравствуйте, [Имя]! Как самочувствие после приёма 
         у доктора Мадякина? [Вопросы про боль / состояние]"
    ↓
    Отправка через Telegram
    ↓
    Сбор ответа (оценка 1-10 + текстовый фидбэк)
    ↓
    IF (оценка ≥ 9):
        ↓
        Просьба оставить отзыв (Я.Карты / 2ГИС)
    ELSE IF (оценка ≤ 6):
        ↓
        Алерт руководству + звонок администратора
    ↓
    Рассказ о последствиях (контекстно под диагноз):
        "Это нормально:..."
        "Обратите внимание:..."
    ↓
    IF (был разовый приём):
        ↓
        Предложение курса → Запись
    ↓
    UPDATE appointments SET followup_sent = TRUE, nps_score = [оценка]
```

#### Workflow 4: Возврат no-show

```
Триггер: Запланированная задача (каждые 15 минут)
    ↓
Запрос к PostgreSQL:
    SELECT * FROM appointments
    WHERE status = 'scheduled'
    AND datetime < NOW() - INTERVAL '1 hour'
    AND patient_arrived = FALSE
    AND noshow_contacted = FALSE
    ↓
Для каждого no-show:
    ↓
    Отправка через Гермеса:
        "Вы были записаны на [время], но не смогли прийти. 
         Всё в порядке? Перенести?"
    ↓
    Ожидание ответа (30 минут)
    ↓
    IF (ответил + хочет перенести):
        ↓
        Поиск свободного времени → Новая запись
    ELSE IF (ответил + не хочет):
        ↓
        UPDATE appointments SET status = 'cancelled'
    ELSE (не ответил):
        ↓
        Задача администратору: "Позвонить [Имя]"
    ↓
    UPDATE appointments SET noshow_contacted = TRUE
```

#### Workflow 5: Реактивация "спящих"

```
Триггер: Запланированная задача (раз в неделю, понедельник 10:00)
    ↓
Запрос к PostgreSQL (RFM-сегментация):
    -- RiskWhale (отвалились 6-12 месяцев)
    SELECT * FROM patients
    WHERE last_visit BETWEEN NOW() - INTERVAL '12 months' AND NOW() - INTERVAL '6 months'
    AND ltv > 50000
    AND reactivation_attempts < 3
    
    -- SleepingWhale (1-2 года)
    SELECT * FROM patients
    WHERE last_visit BETWEEN NOW() - INTERVAL '2 years' AND NOW() - INTERVAL '1 year'
    AND ltv > 50000
    AND reactivation_attempts < 2
    
    -- LostWhale (2+ года)
    SELECT * FROM patients
    WHERE last_visit < NOW() - INTERVAL '2 years'
    AND ltv > 100000
    AND reactivation_attempts < 1
    ↓
Для каждого сегмента → персонализированное сообщение:
    ↓
    RiskWhale:
        "Давно не виделись! Бесплатная контрольная диагностика"
    SleepingWhale:
        "Прошло больше года... Бесплатный осмотр + видеоразбор"
    LostWhale:
        "Помните, как мы работали над [диагноз]? Хотим узнать, как сейчас"
    ↓
Отправка через Гермеса (Telegram / WhatsApp)
    ↓
Ожидание ответа (7 дней)
    ↓
IF (ответил + записался):
    ↓
    UPDATE patients SET reactivated = TRUE, reactivation_date = NOW()
ELSE IF (ответил + отказался):
    ↓
    UPDATE patients SET reactivation_attempts = reactivation_attempts + 1
ELSE (не ответил):
    ↓
    Retry через 2 недели (если attempts < limit)
```

#### Workflow 6: Поздравления с ДР + акция

```
Триггер: Запланированная задача (каждый день 9:00)
    ↓
Запрос к PostgreSQL:
    SELECT * FROM patients
    WHERE EXTRACT(MONTH FROM birthdate) = EXTRACT(MONTH FROM CURRENT_DATE)
    AND EXTRACT(DAY FROM birthdate) = EXTRACT(DAY FROM CURRENT_DATE)
    AND birthday_greeted_year != EXTRACT(YEAR FROM CURRENT_DATE)
    ↓
Для каждого пациента:
    ↓
    Генерация персонализированного поздравления:
        "С днём рождения, [Имя]! 🎉
         Дарим 20% скидку на любую услугу до конца месяца.
         Записаться?"
    ↓
    Отправка через Гермеса
    ↓
    IF (пациент хочет записаться):
        ↓
        Запись + применение промокода в 1С
    ↓
    UPDATE patients SET birthday_greeted_year = CURRENT_YEAR
```

#### Workflow 7: Сбор отзывов

```
Триггер: Запланированная задача (каждый день 18:00)
    ↓
Запрос к PostgreSQL:
    SELECT * FROM appointments
    WHERE status = 'course_completed'
    AND completed_at BETWEEN NOW() - INTERVAL '3 days' AND NOW() - INTERVAL '2 days'
    AND nps_score >= 9
    AND review_requested = FALSE
    ↓
Для каждого пациента:
    ↓
    Отправка через Гермеса:
        "Спасибо за доверие! Если вам всё понравилось, 
         могли бы оставить отзыв на [Я.Карты / 2ГИС / ProDoctors]?
         Вот ссылка: [url]"
    ↓
    Ожидание (3 дня)
    ↓
    Проверка: оставил ли отзыв (парсинг страницы)
    ↓
    IF (оставил):
        ↓
        Благодарность + бонус (10% на следующий курс)
        ↓
        UPDATE patients SET review_left = TRUE
    ↓
    UPDATE appointments SET review_requested = TRUE
```

### 8.1. Безопасность

#### Принцип "Defence in Depth" (многослойная защита)

**Слой 1: Сеть**
- Файрвол на всех серверах (UFW / iptables)
- Только необходимые порты открыты наружу
- VPN для доступа к админ-панелям (Wireguard)
- DDoS-защита (Cloudflare / путём провайдера)

**Слой 2: Аутентификация и авторизация**
- JWT токены для API (с коротким TTL + refresh tokens)
- RBAC (Role-Based Access Control) для пользователей
- 2FA для администраторов (TOTP)
- Rate limiting на все публичные эндпоинты

**Слой 3: Данные**
- Шифрование паролей (bcrypt, cost factor 12)
- Шифрование чувствительных данных в БД (pgcrypto)
- Шифрование в transit (TLS 1.3)
- Шифрование at rest (LUKS для дисков с данными)

**Слой 4: Приложение**
- Input validation на все входные данные
- Prepared statements для SQL (защита от SQL injection)
- CORS правильно настроен
- CSP (Content Security Policy) для веб-интерфейсов
- Регулярные обновления зависимостей (Dependabot)

**Слой 5: Мониторинг и реагирование**
- Логирование всех действий
- Алерты при подозрительной активности
- Регулярные аудиты безопасности
- План реагирования на инциденты

### 8.3. Отказоустойчивость

#### High Availability (HA) для критических компонентов

**PostgreSQL: Master-Replica**
```
┌─────────────┐
│  Master     │  ← Все записи
│  (primary)  │
└──────┬──────┘
       │ Streaming replication
       │
       ▼
┌─────────────┐
│  Replica    │  ← Только чтение (для аналитики)
│  (standby)  │
└─────────────┘

Автоматический failover через Patroni / repmgr
```

**Redis: Sentinel**
```
┌──────────┐      ┌──────────┐      ┌──────────┐
│ Master   │◄────►│ Replica1 │◄────►│ Replica2 │
└──────────┘      └──────────┘      └──────────┘
     ▲                  ▲                  ▲
     │                  │                  │
     └──────────────────┴──────────────────┘
                 Sentinel (мониторинг)
                 
При падении master → автоматическое промоушен replica
```

**MinIO: Erasure Coding**
```
Файл делится на N частей, из которых достаточно M для восстановления
Пример: 4+2 (4 data + 2 parity)
Можно потерять 2 диска из 6 без потери данных
```

**Load Balancing для API:**
```
        ┌──────────┐
        │  Nginx   │  ← Load Balancer
        └────┬─────┘
             │
    ┌────────┼────────┐
    │        │        │
    ▼        ▼        ▼
┌──────┐ ┌──────┐ ┌──────┐
│ API1 │ │ API2 │ │ API3 │  ← Python API instances
└──────┘ └──────┘ └──────┘

Health checks каждые 5 секунд
При падении одного → трафик на оставшиеся
```

#### Circuit Breaker Pattern

Для внешних интеграций (1С, VoIP, Telegram API):

```python
from circuitbreaker import circuit

@circuit(failure_threshold=5, recovery_timeout=60)
def call_1c_api(endpoint, data):
    """
    Если 5 подряд неудачных запросов → circuit открывается
    Все последующие запросы немедленно падают с исключением
    Через 60 секунд → попытка восстановления
    """
    response = requests.post(f"{1C_BASE_URL}/{endpoint}", json=data)
    response.raise_for_status()
    return response.json()
```

## Приложения

### A. Схема базы данных (основные таблицы)

```sql
-- Пациенты (синхронизированы с 1С)
CREATE TABLE patients (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    external_id UUID,  -- ID из 1С
    first_name VARCHAR(100) NOT NULL,
    last_name VARCHAR(100) NOT NULL,
    middle_name VARCHAR(100),
    birthdate DATE,
    gender VARCHAR(10),
    phone VARCHAR(20),
    email VARCHAR(100),
    telegram_id BIGINT,
    
    -- RFM-сегментация
    rfm_segment VARCHAR(50),
    ltv DECIMAL(10, 2),
    last_visit TIMESTAMP,
    total_visits INT DEFAULT 0,
    
    -- Статусы
    status VARCHAR(20) DEFAULT 'active',
    
    -- Метаданные
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP,
    synced_at TIMESTAMP,
    
    UNIQUE(external_id)
);

-- Записи на приём
CREATE TABLE appointments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    external_id UUID,  -- ID из 1С
    patient_id UUID REFERENCES patients(id),
    doctor_id UUID REFERENCES doctors(id),
    
    datetime TIMESTAMP NOT NULL,
    duration_minutes INT DEFAULT 60,
    service_type VARCHAR(100),
    
    -- Статусы
    status VARCHAR(20) DEFAULT 'scheduled',  -- scheduled, completed, cancelled, noshow
    confirmed BOOLEAN DEFAULT FALSE,
    confirmed_at TIMESTAMP,
    patient_arrived BOOLEAN DEFAULT FALSE,
    
    -- Воронка
    followup_sent BOOLEAN DEFAULT FALSE,
    followup_sent_at TIMESTAMP,
    nps_score INT,
    nps_comment TEXT,
    closed_to_course BOOLEAN DEFAULT FALSE,
    
    -- Обработка no-show
    noshow_contacted BOOLEAN DEFAULT FALSE,
    noshow_contacted_at TIMESTAMP,
    
    -- Метаданные
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP,
    synced_at TIMESTAMP,
    
    UNIQUE(external_id)
);

-- Врачи
CREATE TABLE doctors (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    external_id UUID,  -- ID из 1С
    first_name VARCHAR(100) NOT NULL,
    last_name VARCHAR(100) NOT NULL,
    middle_name VARCHAR(100),
    specialization VARCHAR(100),
    
    -- Метаданные
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP,
    
    UNIQUE(external_id)
);

-- Сообщения Гермеса (история диалогов)
CREATE TABLE germes_messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID NOT NULL,
    user_id UUID,  -- patient_id или employee_id
    user_type VARCHAR(20),  -- 'patient' или 'employee'
    
    role VARCHAR(20) NOT NULL,  -- 'user', 'assistant', 'system'
    content TEXT NOT NULL,
    
    -- Контекст
    intent VARCHAR(50),
    entities JSONB,
    
    -- Телефония (если голосовой звонок)
    is_voice BOOLEAN DEFAULT FALSE,
    audio_url VARCHAR(500),
    transcription TEXT,
    
    -- Метаданные
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_germes_messages_session ON germes_messages(session_id);
CREATE INDEX idx_germes_messages_user ON germes_messages(user_id);
CREATE INDEX idx_germes_messages_created ON germes_messages(created_at DESC);

-- Задачи для сотрудников
CREATE TABLE tasks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    type VARCHAR(50) NOT NULL,  -- 'call_patient', 'followup', 'document_review'...
    assigned_to UUID REFERENCES employees(id),
    patient_id UUID REFERENCES patients(id),
    
    title VARCHAR(255) NOT NULL,
    description TEXT,
    
    status VARCHAR(20) DEFAULT 'pending',  -- pending, in_progress, completed, cancelled
    priority VARCHAR(20) DEFAULT 'medium',  -- low, medium, high, urgent
    
    due_date TIMESTAMP,
    
    -- Метаданные
    created_at TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP,
    
    completed_by UUID REFERENCES employees(id)
);

-- И другие таблицы...
```
